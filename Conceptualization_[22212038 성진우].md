# 1. Business Purpose

 인간은 사회적 동물이다’라는 아리스토텔레스의 말처럼 인간은 타인과 관계를 맺고 공동체를 이루며 살아가는 존재이다. 현대사회에서 사람들은 가족, 친구, 직장 등 다양한 관계 속에서 살아가며, 최근에는 바쁜 일상 속에서도 취미 기반의 소모임을 통해 스트레스를 해소하려는 경향이 증가하고 있다. 특히, 최근 주목받고 있는 ‘감자튀김 모임’과 같은 새로운 형태의 소모임 문화는 짧고 부담 없는 만남을 통해 2030 세대에게 큰 관심을 받고 있다.
 이러한 흐름에 따라 취향 공유 플랫폼 또한 빠르게 성장하고 있다. 대표적으로 당근마켓은 기존 중고거래 중심 서비스에서 지역 기반 커뮤니티로 확장하며 ‘모임’ 기능을 도입하였고, 출시 1년 만에 이용자 수 1500만 명을 돌파하였다. 모임 개설 수와 가입자 수 또한 크게 증가하며 현대 사회에서 모임이 중요한 사회적 활동으로 자리잡고 있음을 보여준다.
 그러나 대학 환경에서는 상황이 다소 다르다. 대학에서의 모임은 단순한 취미 활동을 넘어 조별과제, 공모전, 종합설계 프로젝트 등 평가와 직결되는 경우가 많다. 이러한 팀 프로젝트는 협업 능력, 의사소통 능력, 문제 해결 능력을 기를 수 있는 중요한 경험이지만, 동시에 높은 피로도를 유발하는 요소로 작용한다. 특히 학업, 아르바이트, 개인 일정 등으로 바쁜 대학생들에게는 팀원 모집과 일정 조율 자체가 큰 부담으로 작용한다.
 실제로 많은 학생들이 팀 프로젝트의 중요성을 인식하면서도 팀 구성의 어려움으로 인해 참여를 꺼리는 경우가 많다. 또한 관련 정보를 쉽게 접하지 못하거나, 팀원을 구하기 위한 적절한 플랫폼이 부족하다는 문제도 존재한다. 기존의 익명 커뮤니티는 다양한 주제가 혼재되어 있어 팀원 모집에 적합하지 않으며, 익명성으로 인해 지원자의 정보(기술 스택, 경험 등)를 별도로 확인해야 하는 번거로움이 발생한다. 특히 컴퓨터공학 분야에서는 사용 언어, 개발 경험, 기술 스택 등 고려해야 할 요소가 많아 이러한 문제는 더욱 두드러진다.
 이러한 문제를 해결하기 위해 대학생을 대상으로 한 팀원 매칭 및 팀 운영 지원 플랫폼 **‘Creation Time’** 를 개발하고자 한다본 시스템의 목표는 **‘효율적인 팀원 모집’** 과 **‘원활한 팀 운영’** 이다. 사용자는 원하는 주제로 팀을 생성할 수 있으며, 팀 생성 시 자동으로 팀 리더 권한이 부여된다. 이후 팀 리더는 모집글을 작성하고, 지원자들의 정보를 체계적으로 확인한 뒤 간단한 조작을 통해 승인 또는 거절을 수행할 수 있다.팀이 구성된 이후에는 팀 페이지를 중심으로 다양한 협업 기능이 제공된다. 팀원 목록 확인, 일정 등록 및 공유, 투표를 통한 의사결정, 게시판을 통한 정보 공유 및 의사소통 기능을 통해 팀 운영을 효율적으로 지원한다. 또한 팀 리더는 팀원 관리 기능을 통해 팀 구성을 조정할 수 있으며, 투표 결과 및 지원자 현황을 파일 형태로 내보내어 관리 효율성을 높일 수 있다.
  본 시스템은 단순한 모임 플랫폼을 넘어, 대학생들이 팀 프로젝트를 보다 효율적으로 수행할 수 있도록 지원하는 것을 목표로 한다.

# 3. Use Case List

> User는 Member와 TeamLeader로 구분된다.

## Use Case Table

이장은 프로젝트에서 사용될 use case list를 간략하게 설명하였다. User는 역할에 따라 Member와 TeamLeader로 구분된다

### 1) SignUp

| Actor | Description |
|------|-------------|
| Member | 사용자는 시스템을 이용하기 위해 회원가입을 수행한다. 사용자는 이름, 이메일 등 기본 정보와 필요에 따른 기술 스택에 관한 정보를 입력한다. 회원가입시에는 모든 사용자는 Member로 지정된다. |

---

### 2) Login

| Actor | Description |
|------|-------------|
| Member, TeamLeader | 사용자가 Id, 비밀번호를 통하여 사용자임을 인증하고, 본인의 계정으로 로그인한다. TeamLeader의 경우 팀관리 기능을 추가로 사용할 수 있다. |

---

### 3) CreateTeam

| Actor | Description |
|------|-------------|
| Member | 사용자가 팀을 만든다. 팀의 이름, 설명, 모집 요건들을 입력하며, 팀 창설시 해당 사용자는 TeamLeader권한을 부여받는다. |

---

### 4) CreateRecruitPost

| Actor | Description |
|------|-------------|
| TeamLeader | 팀원 모집을 위한 게시글을 작성한다. 팀의 목적, 요구조건, 활동계획 등의 정보가 포함되어야 한다. 해당 기능은 CreateTeam이후 TeamLeader여야만 가능하다. 작성된 글은 모집글 게시판에 등록된다. |

---

### 5) ApplyTeam

| Actor | Description |
|------|-------------|
| Member | 팀원 모집 글을 확인하고, 관심있는 팀에 지원한다. |

---

### 6) ReviewApplication

| Actor | Description |
|------|-------------|
| TeamLeader | 팀에 들어온 신청 목록을 확인하고 신청을 수락한다. 각 지원자에 대해 세부 정보를 확인할 수 있으며 승인 또는 거절을 통해 팀원을 선정할 수 있다. 승인된 사용자는 팀페이지에 접근 할 수 있다. |

---

### 7) ManageTeam

| Actor | Description |
|------|-------------|
| TeamLeader | 팀 리더는 팀원 관리 기능을 통해 팀원을 제외하거나 역할을 부여하는 등 팀 구성을 관리할 수 있다. |

---

### 8) CreateVote

| Actor | Description |
|------|-------------|
| TeamLeader | 원활한 의사결정을 위해 다양한 투표를 만들 수 있다. 투표에 누가 참여했는지 확인이 가능하다. |

---

### 9) ParticipateVote

| Actor | Description |
|------|-------------|
| Member | 팀 리더가 올린 투표에 참여 가능하다. 투표는 설정된 시간내에 수정가능하며, 투표시 항목별 투표현황을 확인 할 수 있다 |

---

### 10) ViewVoteResult

| Actor | Description |
|------|-------------|
| Member, TeamLeader | 이전에 진행된 투표 결과를 확인할 수 있다. 확인은 가능하지만, 수정은 불가능하다. |

---

### 11) CreateSchedule

| Actor | Description |
|------|-------------|
| TeamLeader | 팀페이지에 존재하는 캘린더에 일정을 등록한다. 캘린더에 라인 형태로 표시되며, 하단부에 정리된 형태로 표시한다. |

---

### 12) WritePost

| Actor | Description |
|------|-------------|
| Member, TeamLeader | 팀페이지에서 글을 작성할 수 있다. |

---

### 13) ReadPost

| Actor | Description |
|------|-------------|
| Member, TeamLeader | 다른 사용자가 작성한 글을 확인하여 팀 활동과 관련된 정보를 공유받고, 게시글을 통해 팀원 간 의사소통을 할 수 있다 |

---

### 14) ExportData

| Actor | Description |
|------|-------------|
| Team Leader | 팀 신청 명단, 투표 결과등을 csv 파일 형식으로 다운로드 받을 수 있다. |

# 4. Concept of Operation

## 1) SignUp

| 항목 | 내용 |
|------|------|
| Purpose | 사용자가 시스템을 이용하기 위한 계정을 생성할 수 있도록 한다. |
| Approach | 사용자는 이름, ID, 비밀번호, 이메일 등 기본 정보와 희망하는 팀의 주제, 스택등을 입력하여 회원가입을 요청하고, 시스템은 이를 데이터베이스에 저장한다. |
| Dynamics | 사용자가 회원가입 화면에서 정보를 입력하고 제출할 경우 |
| Goals | 사용자 계정을 생성하여 시스템을 사용할 수 있다. 서버는 사용자의 정보를 받아 저장하여 다른 Operation에 활용될 정보를 수집한다. |

## 2) Login

| 항목 | 내용 |
|------|------|
| Purpose | 등록된 사용자의 인증을 통해 시스템에 접근할 수 있도록 한다. |
| Approach | 사용자가 ID와 비밀번호를 입력하면 시스템은 데이터베이스를 탐색하여 동일한 정보를 가진 인원을 확인한다. 둘중 하나라도 틀리면 오류 메시지를 출력한다. |
| Dynamics | 사용자가 로그인 버튼을 누를 경우 |
| Goals | Id와 비밀번호를 통해 사용자가 자신을 인증하여 자신에게 주어진 권한을 휙득한다. |

## 3) CreateTeam

| 항목 | 내용 |
|------|------|
| Purpose | 사용자가 새로운 팀을 생성할 수 있도록 한다. |
| Approach | 사용자는 팀 정보를 입력하고 팀 생성 버튼을 누르면 시스템은 이를 데이터베이스에 저장하고 요청한 사용자에게 팀 리더 권한을 부여한다. |
| Dynamics | 사용자가 팀 생성 요청을 수행할 경우 |
| Goals | Member에서 시스템으로부터 권한을 부여받아 TeamLeader로 승격된다. 최소 활동 단위인 Team을 형성한다. |

## 4) CreateRecruitPost

| 항목 | 내용 |
|------|------|
| Purpose | 팀 리더가 팀원 모집을 위한 게시글을 작성할 수 있도록 한다. |
| Approach | 팀 리더는 모집 관련 정보를 입력하여 게시글을 작성한다. |
| Dynamics | 팀 생성 이후 팀 리더가 팀원을 모으기 시작할 때, 팀원 모집글을 작성할 때 |
| Goals | 모집 글을 작성하여 모집글 게시판에 글이 올라오고, 다른 사용자가 팀에 가입할 수 있게 된다. |

## 5) ApplyTeam

| 항목 | 내용 |
|------|------|
| Purpose | 사용자가 관심 있는 팀에 지원할 수 있도록 한다. |
| Approach | 사용자는 모집글을 확인하고 모집글 하단부에 있는 버튼을 눌러 해당 팀에 진원할 수 있다. |
| Dynamics | 사용자가 팀 지원 버튼을 누를 경우 |
| Goals | 사용자가 신청 팀의 지원자 목록에 추가된다. |

## 6) ApplyTeam

| 항목 | 내용 |
|------|------|
| Purpose | 팀 리더가 팀 지원자를 승인/거절 할 수 있도록 한다. |
| Approach | 팀 리더는 지원 목록과 각 지원자의 기본 정보 및 희망 주제를 확인할 수 있다. 팀리더가 승인 또는 거절을 선택하며 승인을 누를 경우 팀 정보에 팀원을 추가한다. |
| Dynamics | 팀 리더가 지원 목록을 확인할 경우 |
| Goals | Team Leader가 승인/거절을 통해 팀원을 선정, 팀원이 팀 정보에 추가된다 |

## 7) ManageTeam

| 항목 | 내용 |
|------|------|
| Purpose | 팀 리더가 팀 구성 및 역할을 관리할 수 있도록 한다. |
| Approach | 팀 리더는 기존에 입력한 팀 정보를 바꾸거나 팀에 소속된 팀원 명단을 확인할 수 있다. 팀원을 팀에서 제외하거나 역할을 부여하는 작업을 수행하고 시스템은 팀 정보에 해당 변경을 반영한다. |
| Dynamics | 팀 리더가 팀 관리 작업을 수행할 경우 |
| Goals | 팀 편성 및 팀 정보가 수정되거나 팀원들에게 역할을 지정하여 서로의 역할을 확인 할 수 있다. |

## 8) CreateVote

| 항목 | 내용 |
|------|------|
| Purpose | 팀 리더가 의사결정을 위한 투표를 생성할 수 있도록 한다. |
| Approach | 팀 리더는 투표 정보를 입력하여 투표를 생성한다. |
| Dynamics | 팀 리더가 투표를 생성할 경우 |
| Goals | 팀 리더가 안건에 대하여 투표를 생성 할 수 있다. |

## 9) ParticipateVote

| 항목 | 내용 |
|------|------|
| Purpose | 팀원이 투표에 참여할 수 있도록 한다 |
| Approach | 팀원은 팀리더가 생성한 투표를 확인하고 투표를 참여 할 수 있다. 투표이후 현재 투표 현황을 확인할 수 있으며 재투표 할 수 있다. |
| Dynamics | 팀원이 투표에 참여할 경우 |
| Goals | 사용자가 투표하여 투표 결과가 새롭게 갱신되며 투표 현황을 확인 할 수 있다. |

## 10) ReadVoteResult

| 항목 | 내용 |
|------|------|
| Purpose | 투표 결과를 확인할 수 있도록 한다. |
| Approach | 사용자는 투표 결과를 조회하고 시스템은 선택지별 결과를 제공한다. |
| Dynamics | 사용자가 투표 결과 조회할 경우 |
| Goals | 사용자들이 항목별 투표 결과를 확인 할 수 있다. |

## 11) CreateSchedules

| 항목 | 내용 |
|------|------|
| Purpose | 팀 리더가 팀 일정을 등록할 수 있도록 한다. |
| Approach | 팀 리더는 팀 페이지에 존재하는 캘린더에 일정을 눌러 추가할 수 있다. 추가시 캘린더에 선 형태로 표시되며, 캘린더 하단부에서 요약된 일정을 확인할 수 있다. |
| Dynamics | 팀 리더가 일정을 등록할 경우 |
| Goals | 사용자들이 일정을 한눈에 확인할 수 있다. |

## 12) WritePost

| 항목 | 내용 |

# 5. Problem Statement

## OverViwe
Creation Time은 웹 기반의 팀원 매칭 및 일정 관리 프로그램이다. 팀리더가 팀 형성 후, 팀원 모집글을 쓰면 사용자가 신청하여 팀을 형성한다. 팀원 모집 이후에는 시간 결정등 팀 프로젝트 진행에서 일정 부드러운 일정조율을 위한 시간투표, 일정등록, 글작성등의 기능을 지원한다.

### Problem #1 – 직관적인 인터페이스
  본 시스템은 팀 페이지를 중심으로 다양한 기능과 정보가 동시에 제공되기 때문에 사용자에게 많은 정보가 한 번에 노출될 수 있다. 이로 인해 사용자가 원하는 정보를 찾거나 기능에 접근하는 데 어려움이 발생할 수 있으므로, 직관적인 정보 구조와 사용자 인터페이스 설계가 필요하다.

### Problem #2 – 다양한 디바이스 환경
  사용자는 모바일 기기와 데스크톱 환경 등 다양한 디바이스를 통해 시스템에 접근할 수 있다. 특히 간단한 조회는 모바일에서, 팀 운영과 같은 복잡한 작업은 데스크톱 환경에서 이루어질 가능성이 높다. 따라서 디바이스 특성에 맞는 사용자 인터페이스를 제공할 수 있는 설계가 필요하다.


### Probelm #3 – 모임 구분
 공모전, 스터디는 다양한 분야가 존재한다. 전공, 관심사에 따라 만들어질 팀의 종류는 매우 다양하다. 해당 항목들을 효과적으로 정리할 수 있는 분류가 필요하다. Problem #4 – 데이터 무결성과 일관성
본 시스템은 사용자, 팀, 투표 등 다양한 데이터를 저장하며, 이들 간의 관계가 복잡하게 연결되어 있다. 특히 팀 지원 및 승인 과정에서 동시에 여러 요청이 발생할 경우 데이터의 일관성이 깨질 수 있으며, 모집 상태, 지원 상태, 투표 상태 등 다양한 상태 변화 또한 체계적으로 관리되어야 한다. 따라서 데이터의 무결성과 일관성을 유지할 수 있는 저장 구조 설계가 필요하다.

### Problem #5 – 권한 및 역할 관리
팀 프로젝트에서는 팀 리더와 일반 팀원 간의 역할이 구분된다. 팀원 모집 승인, 일정 생성, 데이터 관리 등 특정 기능은 제한된 사용자만 수행해야 한다. 따라서 사용자 권한을 구분하고 이에 따라 기능 접근을 제어하는 설계가 필요하다.

### Problem #6 – 동시 접속 처리
  본 시스템은 팀 프로젝트를 위한 플랫폼으로, 다수의 사용자가 동시에 접속할 가능성이 높다. 따라서 여러 사용자의 요청을 안정적으로 처리할 수 있도록 동시 접근에 대한 고려가 필요하다.
