# 1. Introduction

### 1) Summary
Creation Time은 대학생을 대상으로 한 학술적 팀 형성 및 팀 운영 지원 웹 서비스이다.  
본 시스템은 사용자가 원하는 주제로 팀을 만들고, 팀원 모집글을 통해 지원을 받고, 승인과 거절을 거쳐 팀을 형성한 뒤, 이후에는 게시글, 댓글, 투표, 일정, CSV 내보내기 기능을 통해 팀 운영을 지원하는 것을 목표로 한다.  
본 Design 문서는 이러한 요구사항을 실제 구현 구조로 구체화하기 위해 작성되었다.

### 2) Important points of design
본 시스템의 가장 중요한 설계 원칙은 풍부한 도메인 모델(Rich Domain Model)을 채택하는 것이다.  
Entity는 단순히 데이터를 저장하는 객체가 아니라 상태와 행동을 함께 가지는 도메인 객체로 설계하였으며, 비즈니스 규칙은 가능한 한 Entity 내부에 배치하였다.  
Service는 여러 Entity가 협력하는 흐름을 조정하고, Repository는 영속성 처리를 담당하며, Controller는 요청의 진입점 역할을 수행한다.

또한 도메인의 책임을 명확히 하기 위해 주요 객체를 분리하였다. 특히 권한 검증은 특정 도메인에 집중시키지 않고, 각 도메인 객체가 자신의 상태와 규칙에 따라 독립적으로 수행하도록 설계하였다.  

도메인 객체에서는 조회와 검증을 분리하였다. 조회(Query)는 `is*`, `has*`, `get*` 계열 메서드를 통해 수행되며, 도메인의 상태를 변경하지 않고 값을 반환하여 부작용없이 안전하게 사용할 수 있도록 설계하였다.
반면 검증은 동작을 수행하기 전에 사전조건을 확인하기 위한 역할이다. `assert` 계열 메서드를 사용하며 조건을 만족하지 않을 경우 즉시 예외를 발생시킴으로써 잘못된 상태 변경을 사전에 차단하고, 오류 상황을 명확하게 표현할 수 있도록 하였다.
검증을 상태 변경 로직 내부에 포함하지 않고 별도의 메서드로 분리함으로써, Service 계층에서 각 도메인 객체에 actor를 전달하여 검증을 수행, 해당 검증을 기반으로 상태 변경을 호출하는 방식으로 동작한다. 이를 통해 도메인은 자신의 규칙을 스스로 보호하고 캡슝성을 유지하도록 하였으며, Service는 흐름을 조정하는 역할에 집중함으로써 계층 간 책임을 명확히 분리하였다.

삭제는 Soft Delete가 아니라 Hard Delete를 기본으로 하며, DTO는 Controller와 Service 사이의 데이터 전달용으로만 사용하여 도메인 객체가 외부에 직접 노출되지 않도록 설계하였다.  

마지막으로 팀 단위 기능은 다수의 사용자가 동시에 접근할 수 있으므로,  
Service 계층에서 트랜잭션 경계를 명확히 설정하고, 필요한 경우 락을 통해 동시성 문제를 방지하도록 설계하였다.

## 2.1 Domain layer
도메인 계층은 시스템의 핵심 비즈니스 규칙을 정의하는 계층으로, 각 Entity는 단순한 데이터 저장 객체가 아니라 상태와 행위를 함께 가지는 도메인 객체로 설계하였다.

풍부한 도메인 모델(Rich Domain Model)을 기반으로 하며 비즈니스 로직과 권한 검증은  Entity 내부에 메서드로서 배치하였다. 이를 통해 서비스 계층에 로직이 분산되는 것을 방지하고, 각 도메인 객체가 자신의 상태와 규칙을 스스로 보호하도록 설계하였다.

또한 Post, Vote, Schedule, RecruitPost, Application은 Team에 논리적으로 소속되는 도메인이지만 각각 독립적인 도메인 객체로서  자신의 상태와 생명주기를 관리하도록 설계하였다.

삭제는 도메인 객체의 상태를 변화시키는 행위가 아니라, 영속성 계층에서 객체를 제거하는 시스템 작업이다.따라서 도메인 객체가 직접 삭제를 수행할 경우 Repository에 대한 의존성이 발생하여계층 간 분리가 깨지게 되므로, 삭제는 Service 계층에서 처리하도록 설계하였다.  
대신 삭제 가능 여부는 비즈니스 규칙에 해당하므로, 도메인 객체는 `assertDeletable`과 같은 메서드를 통해 사전조건을 검증하고, Service는 이러한 검증을 기반으로 삭제 흐름을 수행하도록 역할을 분리하였다.

이와 같은 구조를 통해 도메인 규칙의 응집도를 높이고, 계층 간 책임을 명확히 분리하는 것을 목표로 한다.

세부내용에서 속성 및 생성자, getter들은 특별한 의미를 가진것이 아닌 이상 설명에서는 제외하겠다.

### 1) Enum 클래스
도메인 클래스는 아니지만, 도메인에서 사용하는 다양한 EntityClass다.

#### AccountStatus 
사용자의 계정 상태를 나타내는 클래스. Active(정상, 활성화),  RESTRICTED(임시 차단), BANNED(영구정지)로 구성된다.

#### Role
사용자의 권한을 나타내는 클래스. REGISTERED_USER, TEAM_MEMBER,TEAM_LEADER로 구성되어 있는 Actor를 나타내는 클래스다.

이 두 타입은 User가 자신의 접근 권한과 상태를 판단하는 데 사용된다.

### ExportTarget
csv로 추출될 항목을 나타내는 클래스. APPLICATIONS(지원자 명단),MEMBERS(팀원 명단), VOTE_RESULTS(투표들의 결과)로 구성된다.

### ExportTarget
다른 Enum class와는 다르게 Controller에서 사용된다. exportCsv시 추출 대상을 구분하기 위한 클래스.APPLICATIONS(지원자 명단), MEMBERS(팀원 명단), VOTE_RESULTS(투표 결과 모음)으로 구성된다.

### RemovalReason
제명 사유를 나타내는 클래스로 Penalty Score 부과의 기준이 되는 클래스다. 크게 긍정적 사유, 부정적 사유로 구분되며 아래와 같다.
DISOBEYING_INSTRUCTIONS: 지시 불이행 (팀의 규칙이나 방향을 따르지 않음)
EXCESSIVE_DEADLINE_MISSES: 과도한 마감 미준수 (약속한 일정을 자주 어김)
DISRUPTING_TEAM_ATMOSPHERE: 팀 분위기 저해 (갈등을 유발하거나 사기를 떨어뜨림)
PLAN_CHANGE: 계획 변경 (프로젝트 방향이 바뀌어 하차)
MUTUAL_AGREEMENT: 상호 합의 (서로 잘 얘기해서 나가기로 결정)
OPINION_DIFFERENCE: 의견 차이 (생각이 너무 달라 함께하기 어려움)
VOLUNTARY_WITHDRAWAL: 자진 탈퇴 (개인 사정 등으로 스스로 나감)

해당 enum클래스는 메소드와 속성을 가진다. increasesPenalty 속성은  Penalty Score의 증가 여부로, 부정적 사유의 경우 true로 지정한다. increasesPenalty 메서드를 통하여 해당 사유들의 증가 여부를 확인하고, 이에 따라 처리가 진행되게 된다.

### 2) User
User는 시스템 사용자의 기본 정보를 저장하는 핵심 Entity이다. 시스템 사용자의 정체성과 계정 상태, 기술 프로필을 보유하고 있으며User는 자신의 역할과 계정 상태를 스스로 판단하고 변경하는 책임을 가진다.

- `penaltyScore : int` : User별 Penalty Score를 저장하는 속성. 패널티 정책의 기준이 된다.
- `changeAuthority(role : Authority)` : 사용자의 역할을 변경한다. 여기서 역할이란, 사용자간의 역할로 RegisteredUser, TeamMember, TeamLeader를 말한다. 즉, 사용자가 접근 할 수 있는 권한이라고 할 수 있다. 
- `increasePenalty()` : 패널티 점수를 증가시킨다. 패널티 점수를 증가시키고는 제한 여부를 확인하여 
- `decreasePenalty()` : 패널티 점수를 감소시킨다.
- `matchesPassword(password : String) : boolean` : 입력한 비밀번호가 현재 비밀번호와 일치하는지 확인한다.
- `assertCanUseService()` : 현재 계정이 서비스를 사용할 수 있는 상태인지 검증한다. 사용자가 영구정지 혹은 이용 제한 상태인지 검증한다.

- `applyRestriction()` : increasePenalty이후 연속적으로 동작하는 메서드로, Penalty Score를 확인하여 User를 이용제한, 영구정지 상태로 전환한다.

- `unRestrict()` : 메소드 동작시 시간을 기반으로, 이용 제한 기한이 지났다면 이용 제한을 해제하고 `Active`로 전환한다.  

* Penalty 비즈니스 로직 및 도메인 규칙:
    * 패널티 정책: User가 부정적 사유(명령 불복중, 지나친 마감 미준수, 팀 분위기 방해 등)으로 팀에서 제명 당할시 `increasePenalty()`를 통해 Penalty Score 1점씩 증가하며 제명 없이 프로젝트 기한까지 팀 활동시 -1점을 추가한다. Penalty Score는-2점 미만으로 내려갈 수 없다.
    * 자동 정지/제한 분기: 패널티 점수에 따라 계정 상태가 실시간 제어된다. 
        * 2점 이상: 1개월 이용 제한 (`RESTRICTED`)
        * 4점 이상: 3개월 이용 제한 (`RESTRICTED`)
        * 5점 이상: 영구 정지 (`BANNED`)
    * 상태 복귀: `unRestrict()` 호출 시 현재 시간이 `restrictedUntil`을 지났다면 자동으로 상태를 `ACTIVE`로 전환하고 제한 기한을 초기화(`null`)한다. 계정 이용 가능 상태 판단을 외부 서비스가 아닌 유저 객체가 스스로 책임진다.

### 3) Team
Team은 팀 정보를 저장하는 핵심 Entity이며, 팀 운영의 기준이 되는 중심 도메인 객체이다.  
Team은 팀 구성과 팀원 관계를 관리하며, 팀 내 소속 여부와 역할(TeamLeader)을 기반으로 한 권한 판단을 담당한다. 
논리적으로는 시스템의 핵심 도메인이고 다양한 도메인 객체들이 연관되지만, Team이 Aggregate Root가 되어 생성, 관리, 검증을 모두 담당할 경우, 하나의 객체에 과도한 책임이 집중되어 클래스가 비대해지는 문제가 발생한다. 이는 도메인 객체의 응집성을 저하시킬 뿐만 아니라, 트랜잭션 범위를 불필요하게 확대시키며 성능 저하의 원인이 된다.
따라서 Post, Vote, Schedule, RecruitPost, Application과 같은 도메인 객체들을 Team과 분리된 독립적인 Aggregate로 설계하였으며, Team 도메인은 다른 도메인을 포함하지 않고 팀 구성과 팀원 관계 관리에 집중하도록 설계하였다. 단, Team의 정보 이므로 TeamMember List는 속성으로 보유하도록 설계하였다.

- `addMember(user : User, actor : User)` : TeamMember을 추가한다. 정원 초과 여부와 중복 가입 여부를 확인한 뒤 TeamMember를 생성, 팀원 목록인 members에 추가한다.
- `removeMember(member : User, actor : User)` : 팀원 제외를 수행한다.단, TeamLeader는 제외할 수 없다. 이때 제거 사유를 선택하게 되며, 부정적사유를 고를시 Penalty Score점수가 추가된다.
- `changeLeader(newLeader : User, actor : User)` : TeamLeader를 변경한다. 
- `isFull() : boolean` : 팀이 정원에 도달했는지 확인한다.
- `hasMember(user : User) : boolean` : 팀원이 팀원 목록에 존재하는지 확인한다.
- `isMember(user : User) : boolean` : 특정 사용자가 팀원인지 확인한다.
- `isLeader(user : User) : boolean` : 특정 사용자가 TeamLeader인지 확인한다.
- `getMemberCount() : int` : 현재 팀원 수를 반환한다.
- `assertLeader(actor : User)` : 요청자가 TeamLeader인지 검증한다.
- `assertMember(user : User)` : 요청자가 팀원인지 검증한다.
- `dissolve(actor : User)` : 팀 해체를 수행한다. Team삭제시 Team에 논리적으로 종속된 TeamMember, Post, Schedule, Vote, Application, RecruitPost가 모두 삭제된다.

### 4) TeamMember
TeamMember는 User와 Team 사이의 관계를 표현하는 연관 Entity이다.  
팀 내부에서의 역할을 저장하며, 특정 사용자가 특정 팀에 속해 있는지 확인할 수 있다.  
작은 관계 엔티티이므로 핵심은 소속과 역할을 보관하는 것이다.

- ` changeTeamRole(teamRole : String)` : 팀 내부 역할을 변경한다. 팀 내의 역할은 관심분야, 참여대회주제에 따라, 또 TeamLeader의 방식에 따라 다양하게 구분될 수 있다. 따라서, 특정 역할을 정하는게 아니라 String으로 두고, TeamLeader가 작성하게 한다
- `belongsTo(team : Team) : boolean` : 특정 팀 소속인지 확인한다.

### 5) Application
Application은 사용자의 팀 지원 요청을 나타내는 일회성 도메인 객체이다. 지원자 정보는 사용자가 회원가입시 입력한 정보를 바탕으로 시스템이 자동작성한다.
지원이 생성된 이후에는 값이 바뀌지 않으며, 승인 또는 거절 처리 후 삭제된다.

- `Application()` : Application을 생성한다. 생성자이자 팀의 지원을 의미하는 메서드이다.
- `approve()` : 지원을 승인한다. application의 User를 Team의 addMember를 통해 members에 추가하고 application은 삭제한다
- `reject()` : 지원을 거절한다. Team에는 변동이 없고, application은 바로 삭제한다.
- `belongsTo(team : Team) : boolean` : 특정 팀에 대한 지원인지 확인한다.

### 6) Post
Post는 팀 게시글을 저장하는 Entity이다. Post는 Aggregate Root로 Post를 통해서만 Comment 접근이 가능하고, Commnet를 생성/관리한다. 

- `getContent() : String` : 게시글 내용을 반환한다.
- `hasComment() : boolean` : 댓글이 존재하는지 확인한다.
- `addComment(comment : Comment)` : 댓글을 추가한다.
- `removeComment(comment : Comment)` : 댓글을 제거한다.
- `isWrittenBy(user : User) : boolean` : 작성자 여부를 확인한다.
- `belongsTo(team : Team) : boolean` : 특정 팀 게시글인지 확인한다.
- `assertDeletable(actor : User)` : 삭제 가능한지 사전조건을 검증한다. 객체가 자신을 삭제하는것은 
- `modify(title : String, content : String)` : 제목과 내용을 수정한다.

## 7) RecruitPost
RecruitPost는 팀원 모집 게시글을 표현하는 Entity로 Post를 상속한다.  
RecruitPost는 단순 데이터가 아닌 “팀의 모집 상태”를 나타내는 도메인 객체로, 삭제되지 않고 상태 전환을 통해 재사용된다. Team당 하나만 존재하며 이를 기반으로 RecruitPost의 과도한 글 작성을 방지하고, RecruitPost를 TeamMember 모집 여부를 표현하는 객체로 설계하였다.
- `Closed : boolean` : Team의 현재 모집 여부를 표시하는 객체. false일때 모집 중임을 의미한다. 
- `closeRecruitment()` : 모집을 종료한다.
- `openRecruitment()` : 모집을 시작한다.
- `isRecruiting() : boolean` : 현재 모집 중인지 확인한다.
- `modifyRecruitInfo(...)` : 게시글 자체 정보와 모집 정보를 함께 수정한다 
- `belongsTo(team : Team) : boolean` : 특정 팀의 모집글인지 확인한다.

* RecruitPost 비즈니스 로직 및 도메인 규칙:
  * 앞서서 언급한대로 RecruitPost는 단순 데이터가 아닌 “팀의 모집 상태”를 나타내는 도메인 객체이다. 
  * 팀 페이지내 팀원모집글 작성버튼을 눌럿을때, 기존 정보(속성에 저장된 값)가 있다면 불러오고 저장버튼으로 내용 수정, 게시버튼으로 다시 연다
  * 모집 상태는 다음 조건을 만족해야 한다:
  isRecruiting() = !Closed && (deadline 이전)
  * 모집은 다음 조건 중 하나를 만족하면 종료된다:
  1. deadline이 지난 경우  
  2. 팀원이 모두 모집된 경우  

  * 팀원 증가에 따른 종료 판단은 서비스 계층에서 수행된다:
    if (team.isFull())
        → recruitPost.closeRecruitment()
  * 모집 재개 규칙
  모집은 openRecruitment()를 통해 다시 시작할 수 있다. TeamLeader만이 가능하며, TeamLeader가 RecruitPost세부 보기화면에서 모집 재개(게시)버튼을 수동으로 누르는 방식으로만 동작한다.
   단, 팀원이 이미 모두 채워진 상태에서 재개 요청 시 사용자 재확인 필요  
  "인원이 모두 찼는데 정말로 모집을 다시 시작할까요?" 메세지 출력 확인 시 다시 모집을 시작한다

### 8) Comment
Comment는 특정 게시글(Post혹은 RecruitPost)에 종속되는 Entity이다. 
댓글은 별도의 수정 기능 없이 생성과 삭제 중심으로 운영된다.

- `isWrittenBy(user : User) : boolean` : 작성자 여부를 확인한다.

### 9) Vote
Vote는 팀 내부 의사결정을 위한 독립적인 Entity이다. 생성/삭제는 TeamLeader만 가능하다. 서론에서 언급한대로 권한 검증용 메서드만 생성, 검증 흐름 조절은 Service에서 담당한다.
Vote는 단순한 속성들을 가지므로 생성과 삭제 중심으로 운영된다. 투표 참여는 기한 내에만 가능하고, 결과는 종료 이후에도 조회할 수 있다.

- options : List<String> : 투표 항목들을 저장하는 속성이다. 최소 두개이상의 항목을 유지해야한다. 
- voteResults : Map<Long, Integer> : 투표 결과를 저장하는 속성이다. <UserId, 선택한 항목 인덱스> 형태로 저장한다.

- `participate(user : User, selectedOptionIndex : int)` : 투표 참여를 기록한다.
- `showResult() : Map<Long, Integer>` : 투표 결과를 반환한다.
- `isParticipated(user : User) : boolean` : 이미 참여했는지 확인한다.
- `isExpired() : boolean` : 마감 여부를 확인한다.
- `assertNotExpired()` : 마감 전인지 검증한다.
- `assertDeletable(actor : User)` : 사용자가 삭제 가능한지 검증한다. 
- `belongsTo(team : Team) : boolean` : 특정 팀의 투표인지 확인한다.

### 10) Schedule
Schedule은 팀 일정을 저장하는 Entity이다.  
일정은 생성과 삭제 중심으로 운영되며, 일정을 저장하는 간단한 도메인이므로 수정은 제외시켰다.

- `assertDeletable(actor : User)` : 삭제 가능한지 검증한다.
- `belongsTo(team : Team) : boolean` : 특정 팀 일정인지 확인한다.

## 2.2 Controller layer
Controller layer는 사용자의 요청이 시스템에 최초로 진입하는 진입점이다. 사용자 요청을 받아 요청 데이터(Request DTO)를 해석하고, 필요한 PathVariable, RequestBody, Header 값을 Service 계층으로 전달한다. 이후 Service에서 반환한 결과를 Response DTO 형태로 응답한다.

본 프로젝트에서 Controller는 비즈니스 로직을 직접 처리하지 않는다. 팀 생성, 지원 승인, 투표 참여, 패널티 적용, 권한 검증과 같은 핵심 규칙은 Service와 Domain 계층에서 처리하도록 하였으며, Controller는 API 요청과 응답의 경계를 담당하도록 역할을 제한하였다.

Controller는 기능별로 AuthController, TeamController, ApplicationController, RecruitPostController, PostController, VoteController, ScheduleController로 분리하였다. 또한 예외 응답은 ApiExceptionHandler에서 공통적으로 처리하도록 하여, 각 Controller가 예외 변환 로직을 반복해서 가지지 않도록 하였다.

## 2.3 Service layer
Service Layer는 각 도메인 객체들을 조합하여 서비스 흐름을 조절하는 역할을 담당한다. 트랜잭션 범위 내에서 필요한 도메인 객체를 조회하고, 도메인 계층의 메서드를 호출하여 비즈니스 로직을 실행한다.

Service 계층의 기본 흐름은 요청자 검증, 대상 도메인 조회, 도메인 기준 권한 검증, 상태 변경 순서로 구성된다. 권한 검증은 대상 도메인의 상태를 기준으로 수행되므로, 대상 객체 조회 이후 해당 도메인의 `assertMember`, `assertLeader`와 같은 검증 메서드를 호출한다. 이후 검증이 완료된 경우에만 상태 변경 메서드를 호출하여 전체 흐름을 조정한다. 이를 통해 도메인 로직의 응집도를 높이고, 계층 간 책임을 명확히 분리하였다.동시성 제어가 필요한 변경 흐름에서는 대상 객체를 forUpdate로 잠금 조회한 뒤 같은 트랜잭션 안에서 검증과 상태 변경을 수행한다.

또한 동시성 문제가 발생할 수 있는 흐름에서는 Repository의 `findByIdForUpdate()` 기반 조회를 사용한다. 팀원 승인, 팀원 삭제, 팀 해체, 투표 참여, 모집글 상태 변경처럼 동일한 데이터가 동시에 수정될 수 있는 경우에는 Service 계층에서 트랜잭션 경계를 잡고, 잠금 조회 이후 검증과 상태 변경을 같은 흐름 안에서 처리한다.


### 1) AuthService
AuthService는 회원 생성과 인증을 담당하며, 입력값 검증과 로그인 성공 여부 판단을 조정한다.  
이 계층은 시스템의 첫 진입점이므로, 인증 정보가 제대로 전달되는지와 사용자 상태가 유효한지를 먼저 확인하는 역할을 한다.

- `signUp(request : RequestSignUp) : ResponseAuth`  
  회원가입 요청을 받아 사용자 정보를 저장하고, 가입 결과를 반환한다.  
  입력 데이터를 검증하고 중복 가입 여부를 확인한 뒤, 초기 권한(REGISTERED_USER)과 정상 상태(ACTIVE)를 가진 `User` 엔티티를 생성하여 영속화한다. 비밀번호는 passwordEncoder를 사용하여 해시화를 진행한다.

- `login(request : RequestLogin) : ResponseAuth`  
  로그인 요청을 받아 계정을 검증하고, 인증 결과를 반환한다.  
  로그인 ID로 유저를 조회한 뒤, 비밀번호 일치 여부와 현재 서비스 이용 가능 상태(Penalty Score)를 연쇄적으로 검증하여 인증 결과를 반환한다. 제한 상태인 사용자는 `assertCanUseService()` 검증을 통과하지 못하므로 로그인이 차단된다.

### 2) TeamService
TeamService는 팀 생성, 팀 조회, TeamMember 관리, 역할 변경, 팀 해체와 CSV 내보내기를 담당한다.  
팀과 관련된 권한 검증은 Team 도메인을 기준으로 수행되며, 팀 소속 여부 및 역할에 따라 수행 가능한 작업을 제한한다.

- `createTeam(request : RequestCreateTeam, actorId : Long) : ResponseTeam`  
  팀 생성 요청을 받아 새로운 Team 객체를 생성한다. 팀이 생성되면 요청자는 팀 운영 권한을 가진 TeamLeader가 된다. 팀 생성시에는  이미 팀에 소속된 사용자인지, 중복된 팀 이름인지 먼저 확인한다.

- `readMyTeam(actorId : Long) : ResponseTeam`  
  현재 요청자가 소속된 팀을 조회한다.  
  로그인한 유저가 팀에 소속되어 있을 경우 팀 페이지로 이동할 수 있도록 현재 사용자의 팀 정보를 반환한다.

- `readTeam(teamId : Long, actorId : Long) : ResponseTeam`  
  팀의 기본 정보(teamName, description, members 같은 팀의 속성)와 TeamMember 수, TeamLeader 정보를 조회한다.  
  팀 상세 정보는 TeamUser만 확인할 수 있으므로 Team 도메인의 멤버 검증을 수행한다.

- `changeLeader(teamId : Long, request : RequestChangeLeader, actorId : Long) : void`  
  기존 TeamLeader를 새로운 TeamLeader로 변경한다.  
  TeamLeader 교체는 팀 운영 권한과 직접 연결되므로 기존 TeamLeader 권한 검증이 반드시 선행된다. 새 리더가 되는 User는 동시성 문제를 방지하기 위해 잠금 조회한다.

- `changeRole(teamId : Long, memberId : Long, request : RequestChangeRole, actorId : Long) : void`  
  특정 TeamMember의 역할을 수정한다. 팀 역할 수정은 팀 전반에 관한 내용으로 TeamLeader 권한 검증이 필요하다.

- `dismissMember(teamId : Long, memberId : Long, reason : RemovalReason, actorId : Long) : void`  
  TeamMember를 팀에서 제외한다.  
  TeamLeader만 가능하며, 제외 대상이 실제 팀원인지, TeamLeader가 아닌지 확인한 뒤 제명 사유를 기준으로 삭제 흐름을 수행한다. 제명 사유에 따라 사용자 Penalty Score가 변경될 수 있으므로 도메인을 통해 처리한다.

- `exportCsv(teamId : Long, target : ExportTarget, actorId : Long) : Resource`  
  팀 관련 정보를 CSV 파일로 내보낸다.  
  팀원 명단, 지원자 명단, 투표 결과 중 선택한 항목을 CSV 파일로 생성한다. 한글 깨짐을 방지하기 위해 UTF-8으로 인코딩한다.
  CSV 생성은 선택 항목을 기준으로 `exportMembers`, `exportApplications`, `exportVoteResults`로 분리하였다. 각 메서드는 선택된 내보내기 항목에 맞는 데이터를 조회하고, 공통 보조 메서드인 `appendRow`와 `csvValue`를 사용해 CSV 행을 구성한다.  
  `appendRow`는 전달받은 값을 쉼표 기준의 한 행으로 추가하고, `csvValue`는 값에 쉼표, 따옴표, 줄바꿈이 포함된 경우 CSV 규칙에 맞게 따옴표로 감싸도록 처리한다. 투표 결과는 선택지별 득표 수를 행으로 변환해야 하므로 `appendVoteRows`에서 Vote의 결과를 순회하여 공통 행 추가 로직에 전달한다.

  * CSV 내보내기 처리를 위한 보조 메서드
  - `exportMembers(team : Team) : String`
  - `exportApplications(teamId : Long) : String`
  - `exportVoteResults(teamId : Long) : String`
  - `appendVoteRows(csv : StringBuilder, vote : Vote) : void`
  - `appendRow(csv : StringBuilder, values : Object...) : void`
  - `csvValue(value : Object) : String`

- `deleteTeam(teamId : Long, typedTeamName : String, actorId : Long) : void`  
  팀을 해체한다.  
  TeamLeader 권한 검증이 필수적이다. 해체 시에는 동의의 의미로 팀 이름 입력을 받아야 삭제한다. 팀 활동 기한을 넘긴 경우 TeamMember들의 Penalty Score를 낮추고, 권환을 REGISTERED_USER로 되돌린다. 이후 팀에 종속된 지원, 투표, 일정, 댓글, 게시글 데이터를 함께 정리한다.

- `findTeam(teamId : Long) : Team`  
  팀을 조회하기 위한 내부 보조 메서드이다.
  단순 조회 흐름에서 사용한다. 

- `lockTeam(teamId : Long) : Team`  
  팀을 잠금 조회조회하기 위한 내부 보조 메서드이다.
  존재하지 않을 경우 DomainException을 발생시킨다. 팀 구성 변경, 팀 해체, 지원 승인처럼 팀 상태가 변경되는 흐름에서 사용한다.

   `findTeam`,`lockTeam`은 단순히 DB에서 검색하는 단순 조회 메서드다. 조회/수정/삭제등 여러 메서드중에서 기본적으로 흐름이기에 하나의 메서드로 묶어 동일한 조회 및 예외처리를 줄였다.

### 3) ApplicationService
ApplicationService는 팀 지원 흐름을 담당한다.  
Application은 생성 후 변경되지 않는 일회성 요청 객체이므로, 지원을 생성하고 승인 또는 거절 시 제거하는 방식으로 처리한다.

- `applyToTeam(teamId : Long, actorId : Long) : ResponseApplication`  
  사용자가 특정 팀에 지원한다.  
  모집글이 모집 중인지, 이미 팀에 소속되어 있는지, 중복 지원이 아닌지 확인한 뒤 Application을 생성한다. 지원자는 이후 팀에 들어갈 수 있으므로 요청 사용자는 잠금 조회한다.

- `readApplications(teamId : Long, actorId : Long) : List<ResponseApplication>`  
  특정 팀의 지원자 목록을 조회한다. 팀 관리 페이지에서 지원자 목록을 가져올때 사용한다.
  TeamLeader만 확인할 수 있으며, 지원자의 이름, 이메일, 관심 분야, 기술 스택, 희망 역할, Penalty Score 등을 확인할 수 있도록 ResponseApplication으로 변환한다.

- `approveApplication(teamId : Long, applicationId : Long, teamRole : String, actorId : Long) : void`  
  지원을 승인한다.  
   지원서 처리는 TeamLeader만 가능하므로 ActorId를 기반으로 권한 검증을 진행한다. 승인 시 Registerd_User는 TeamMember로 추가되고, 처리된 Application은 제거된다. 팀 정원과 신청자 소속 상태가 동시에 영향을 받을 수 있으므로 팀, 지원서, 신청자 조회에 잠금 조회를 사용한다. 기본적으로 TeamLeader만이 Team관련 조작 및 승인을 할 수 있으나, 예상치 못한 상황으로 인한 요청의 딜레이, 혹은 빠른 속도로 두명 이상이 동시에 승인 될 경우를 대비하여 lock을 사용하였다. 이를 통해 정원 초과 및 중복 혀용을 방지하고자 한다.
   승인 후 팀이 가득 차면 RecruitPost을 자동으로 모집 종료 상태로 변경한다.

- `rejectApplication(teamId : Long, applicationId : Long, actorId : Long) : void`  
  지원을 거절한다.  
  거절 시 Application만 제거되며 팀 구성은 변경되지 않는다. 이 또한 ActorId를 기반으로 TeamLeader 권한 검증을 진행한다.

- `lockApplicationFor(team : Team, applicationId : Long) : Application`  
  특정 Applicant를 잠금 조회하고, 해당 지원서가 대상 팀에 속한 지원서인지 검증한다.  
  
### 4) RecruitPostService
RecruitPostService는 RecruitPost의 저장과 조회, 그리고 모집 시작과 중단을 담당한다.

- `readRecruitPost(teamId : Long) : ResponseRecruitPost`  
  팀의 RecruitPost을 조회한다.  
  RecruitPost가 존재하면 현재 제목, 내용, 모집 정보, 모집 상태를 반환한다.

- `readOpenRecruitPosts() : List<ResponseRecruitPost>`  
  현재 모집 중인 RecruitPost 목록을 조회한다.  
  Registered_User가 로그인 하였을때, RecruitPost목록을 갱신하기 위해 사용한다.

- `saveRecruitPost(teamId : Long, request : RequestRecruitPost, actorId : Long) : ResponseRecruitPost`  
  RecruitPost를 저장하거나 수정한다. 이미RecruitPost가 존재하는 경우 내용을 갱신하고, 존재하지 않는 경우 새로 생성한다.  
  팀당 하나의 RecruitPost만 존재하도록 설계되어 있으므로 생성과 수정이 하나의 흐름으로 처리된다. 모집 인원이 팀 최대 인원을 초과하지 않는지 검증한다.
  현재 시스템은 동시 로그인을 차단하지 못하고 있다. 따라서 TeamLeader만이 접근할 수 있는 기능일지라도 lock을 사용하고 있다.

- `openRecruitment(teamId : Long, confirmFullTeam : boolean, actorId : Long) : void`  
  모집을 시작한다.  
  TeamLeader가 수동 실행만 가능하며, 모집이 가능한 상태인지 확인한 뒤 공개 상태로 전환한다. 팀 인원이 이미 모두 찬 경우에는 TeamLeader만이  경우에만 모집을 다시 시작할 수 있도록 한다.

- `closeRecruitment(teamId : Long, actorId : Long) : void`  
  모집을 중단한다.  
  현재 모집 중인 글을 종료 상태로 바꾸고 RecruitPost게시판 노출을 멈춘다.

- `findByTeam(teamId : Long) : RecruitPost`  
  특정 팀의 RecruitPost를 조회하기 위한 내부 보조 메서드이다.  
  모집글이 존재하지 않으면 DomainException을 발생시킨다.

- `findLockedByTeam(teamId : Long) : RecruitPost`  
  특정 팀의 모집글을 잠금 조회하기 위한 내부 보조 메서드이다.  
  모집 시작과 모집 종료처럼 모집글 상태를 변경하는 흐름에서 사용한다.

- `assertRecruitCountFitsTeam(recruitCount : int, team : Team) : void`  
  모집 인원이 팀의 최대 인원을 넘지 않는지 검증한다.

RecruitPostService에는 Comment 관련 메서드는 물론 기초적인 게시글 관리 메서드를 작성하지 않았다. 도메인 레벨에서 RecruitPost는 Post를 상속하여 구현한다. 따라서 RecruitPostId는 PostID로 취급할 수 있으므로 PostService를 사용할 수 있다. 즉, 공통기능은 PostService로 처리하고, RecruitPost 독립적인 기능만 처리한다.

### 5) PostService
PostService는 팀 내 게시글과 댓글 기능을 담당한다.  
Post과 Comment은 팀 내 의사소통을 위한 핵심 수단이므로, 조회와 수정, 삭제 흐름이 명확하게 분리되어 있다.

RecruitPost는 Post를 상속한다. 이로 인해 메서드의 중복을 피하고 책임을 분리할 수 있었다. 하지만 RecruitPostId는 PostID로 취급할 수 있으므로 RecruitPost가 팀 내 게시글로 처리되고 Post 목록에 저장되는 문제가 발생할 수 있다. 따라서 PostService에서 RecruitPost와의 공통기능을 제외하고는 RecruitPost가 아닌지 검증이 필요하다.

- `createPost(teamId : Long, request : RequestPost, actorId : Long) : ResponsePost`  
  Post을 생성한다.  
  작성자가 TeamMember인지 확인한 뒤 Post을 생성한다.

- `readPosts(teamId : Long, actorId : Long) : List<ResponsePost>`  
  특정 팀의 Post 목록을 조회한다.  
  단순 조회이므로 상태 변경 없이 목록만 반환한다.  RecruitPost가 Post 목록에 섞이지 않도록 Repository 조회 단계에서 Post만 조회한다. 

- `readPost(postId : Long, actorId : Long) : ResponsePost`  
  특정 Post의 상세 내용을 조회한다.  
  Comment와 함께 Post의 세부 정보를 확인할 때 사용한다.

- `modifyPost(postId : Long, request : RequestPost, actorId : Long) : ResponsePost`  
  Post의 제목과 내용을 수정한다.  
  수정 요청을 한 사용자가 Post작성자인지 검증이후 수정이 진행된다.

- `deletePost(postId : Long, actorId : Long) : void`  
  Post을 삭제한다.  
  삭제 시 작성자 여부와 팀 소속 여부를 먼저 검증한 뒤 Hard delete한다.

- `writeComment(postId : Long, request : RequestComment, actorId : Long) : ResponseComment`  
  Post에 Comment을 작성한다.  
  Comment는 Post에 종속되고, 여러 TeamUser가 동시에 접근 가능하다. 따라서, 잠금 조회한 Post에 Commnt을 추가한 뒤 저장된 Comment 정보를 반환한다.

- `deleteComment(postId : Long, commentId : Long, actorId : Long) : void`  
  Comment를 Hard delete한다.  
  Comment는 Post에 종속되므로 게시글과 연결된 흐름 안에서 제거된다. Comment 작성자만 삭제할 수 있다.

- `findTeamPost(postId : Long) : Post`  
  Post을 조회조회하기 위한 내부 보조 메서드이다.
  RecruitPost가 조회된 경우 일반 게시글 API가 아닌 모집글 API를 사용하도록 예외를 발생시킨다.

- `lockTeamPost(postId : Long) : Post`  
  일반 팀 게시글을 잠금 조회하기 위한 내부 보조 메서드이다.
  게시글 수정, 삭제, 댓글 작성, 댓글 삭제처럼 Post 상태가 변경되는 흐름에서 사용한다.

- `assertNotRecruitPost(post : Post) : void`  
  조회된 Post가 RecruitPost인지 검증한다.  
  RecruitPost가 일반 Post API에서 처리되는 것을 방지한다.

### 6) VoteService
VoteService는 Vote 생성, 참여, 결과 조회, 삭제를 담당한다.  
Vote는 TeamLeader만이 생성/삭제가 가능하므로 해당 메서드에서는 권한 검증이 필요하다. 투표 참여는 기한 내에 가능하며, 이미 참여한 사용자는 기한 내에 다시 투표할 수 있도록 기존 선택 값을 갱신하는 방식으로 처리한다.

- `createVote(teamId : Long, request : RequestCreateVote, actorId : Long) : ResponseVote`  
  Vote를 생성한다.  
  TeamLeader만 생성할 수 있으며, 선택지와 마감일이 함께 저장된다.

- `readVotes(teamId : Long, actorId : Long) : List<ResponseVote>`  
  특정 팀의 Vote 목록을 조회한다.  
  단순 조회이므로 상태 변경 없이 목록만 반환한다.

- `readVoteResult(voteId : Long, actorId : Long) : ResponseVoteResult`  
  특정 Vote의 결과를 조회한다.  
  TeamMember만 확인할 수 있으며, 결과는 투표 종료 이후에도 확인할 수 있다.

- `participateVote(voteId : Long, request : RequestParticipateVote, actorId : Long) : void`  
  Vote에 참여한다.  
  투표 대상 Vote를 잠금 조회하고, 팀 소속 여부와 마감 여부를 확인한 뒤 선택한 옵션을 기록한다. 이미 투표한 TeamUser가 다시 참여하면 기존 선택 항목을 새 선택 항목으로 갱신한다.

- `deleteVote(voteId : Long, actorId : Long) : void`  
  Vote를 삭제한다.  
  Team 소속과 TeamLeader 권한을 검증한 뒤 Hard delete한다.

- `findVote(voteId : Long) : Vote`  
  Vote 엔티티를 조회하기 위한 내부 보조 메서드이다.
  Vote ID로 엔티티를 조회하며, 존재하지 않을 경우 DomainException을 발생시킨다..

- `lockVote(voteId : Long) : Vote`  
  특정 Vote를 잠금 조회하기 위한 내부 보조 메서드이다.
  투표 참여와 삭제처럼 Vote 상태가 변경되는 흐름에서 사용한다.

### 7) ScheduleService
ScheduleService는 Schedule 등록, 조회, 삭제를 담당한다.  
Schedule은 생성과 삭제 중심으로 운영되며, 별도의 수정 메서드는 두지 않는다. Schedule도 TeamLeader만이 생성/삭제가 가능하므로 해당 메서드에서는 권한 검증이 필요하다.
Schedule은 단순 생성/삭제 중심의 도메인이지만, 더블클릭이나 중복 요청으로 동일 팀에 일정 생성 요청이 동시에 들어올 수 있다. 또한 팀 해체와 일정 생성이 동시에 수행될 경우 정합성 문제가 발생할 수 있으므로, 생성 시에는 부모 리소스인 Team을 잠금 조회하여 팀 단위의 일정 생성 흐름을 직렬화하였다. 삭제 시에는 기존 Schedule을 잠금 조회하여 동일 일정에 대한 중복 삭제 요청을 방지한다.

- `createSchedule(teamId : Long, request : RequestSchedule, actorId : Long) : ResponseSchedule`  
  TeamLeader가 새 Schedule을 생성하고 저장한다.

- `readSchedules(teamId : Long, actorId : Long) : List<ResponseSchedule>`  
  특정 팀의 Schedule 목록을 조회한다.  
  TeamUser만 조회할 수 있으며, 시작일 기준으로 정렬된 일정을 반환한다.

- `readScheduleDetail(scheduleId : Long, actorId : Long) : ResponseSchedule`  
  Schedule의 상세 내용을 조회한다.  
  기간과 활성 상태를 함께 확인한다.

- `deleteSchedule(scheduleId : Long, actorId : Long) : void`  
  Schedule을 삭제한다.  
  Team 소속과 TeamLeader 권한을 검증한 뒤 Hard delete한다.

- `findSchedule(scheduleId : Long) : Schedule`  
  Schedule을 조회하기 위한 내부 보조 메서드이다.
  존재하지 않을 경우 DomainException을 발생시킨다.

## 2.4 Repository와 DTO

### Repository
Repository는 데이터 저장 및 조회를 담당하는 인터페이스이다.  
UserRepository, TeamRepository, TeamMemberRepository, ApplicationRepository, PostRepository, RecruitPostRepository, CommentRepository, VoteRepository, ScheduleRepository로 구성된다.  
모든 Repository는 저장, 조회, 삭제, 존재 여부 확인을 위한 기본 연산을 제공한다. JPA를 상속하는 방식으로 구현하였기에 interfacetype으로 구현되고, 추상 메소드만 작성하였다. 객체는 의존성 주입 과정에서 자동생성되므로 클래스를 작성하지 않았다.
JPA를 활용하였기에 기본적으로 save, findbyid, findAll등의 메소드들을 제공하고 이외에도
- Team id를 기반으로 한 검색(findbyTeamid)
- 생성 날짜를 기반으로 정렬검색(OrderByCreatedAtDesc)
- 동시성을 위한lock처리 (ForUpdate)
- 팀 id를 기반으로 삭제 (deleteAllByTeamId)
를 추가로 작성하였다. 메서드 이름 기반의 쿼리를 사용하였기에 별도로 설명을 작성하진 않겠다.
예외적으로  RecruitPost는 Post를 상속하여 구현하였다. 이로 인해 메서드의 중복을 피하고 책임을 분리할 수 있었으나 RecruitPostId는 PostID로 취급할 수 있으므로 RecruitPost가 팀 내 게시글로 처리되어 Post 목록에 저장되는 문제가 발생할 수 있다. PostRepository에서는 JPQL기반으로 타입이 Post인지 확인하도록 설계하였다.

### DTO
DTO는 Controller와 Service 계층 간의 데이터 전달을 담당하며, 외부 요청(Request)과 내부 처리 결과(Response)를 표현하기 위한 객체이다.Request DTO는 입력 데이터를 담당하고, Response DTO는 필요한 결과 정보만 전달한다.  
DTO를 사용하는 이유는 도메인 엔티티를 외부에 직접 노출하지 않기 위해 사용한다. 이를 통해 비즈니스 규칙을 보호하고 계층 간 결합도를 낮추며, 내부 구조 변경이 외부 API에 영향을 주지 않도록 하기 위함이다.
DTO는 단순 데이터 전달 역할에 집중할 수 있도록 불변 객체 기반으로 설계하였다. 이를 위해 Record를 활용하였으며, Response DTO의 경우, 보일러플레이트 코드를 줄이고 객체 생성 의도를 명확히 하기 위해 Static Factory Method를 함께 사용하였다.

## 2.5 보조 클래스
Domain들을 직접 처리하는 핵심 클래스는 아니지만, 다른 클래스들에서 많이 사용되거나 공통적으로 사용되는 기능들을 묶어 놓은 클래스이다.

#### 1) ActorService
현재 요청을 수행하는 사용자(Actor)를 검증해서 가져오는 서비스다.

- `requireActor(actorId : Long) : User`  
  요청자가 등록되어 있는지 조회하고 서비스 이용 가능 여부를 검증한 뒤 반환한다.  
  사용자가 존재하는지 확인하고, `assertCanUseService()`를 통해 이용 가능 여부를 판단한다. 단순 조회나 사용자 상태를 변경하지 않는 흐름에서 사용한다.

- `requireActorForUpdate(actorId : Long) : User`  
  요청자를 잠금 조회하고 요청자가 등록 및 서비스 이용 가능 여부를 검증한 뒤 반환한다.  
  기본적으론 requireActor와 동일하나 Repository 조회 시 동시성 처리를 위해 `findByIdForUpdate()`를 사용한다. 사용자 권한, 팀 소속, Penalty Score처럼 User 상태가 변경될 수 있는 흐름에서 사용한다.

#### 2) TextValidator
- `requireText(Value: String, field: String): String`
 입력된 문자열이 null이거나 비어있는지 확인한다. 비어있지 않다면 양쪽의 공백을 제거하고, 비어있다면 해당 value가 어느 field 즉, 어느 속성이었는지를 예외를 통해 전달한다.
 속성의 일관성 있는 관리를 위한 클래스로, 유지보수를 향상시키기 위한 용도이다.

 #### 3) ApiExceptionHandler
 시스템 전역의 예외 처리를 담당하는 클래스이다. Controller 또는 Service 계층에서 발생한 예외를 HTTP 응답으로 변환하는 클래스로, 각 Controller에서 중복 작성하지 않고 전역으로 설정하여시스템 전체의 예외 처리의 일관성을 높이기 위해 설계했다.

- `handleDomainException(exception : DomainException) : ResponseEntity<Map<String, Object>>`
도메인 규칙 위반으로 발생한 DomainException을 처리한다.
권한 부족, 정원 초과, 중복 지원, 잘못된 상태 변경 등 비즈니스 규칙 위반 상황에서 호출되며, HTTP 409(CONFLICT) 상태 코드와 오류 메시지를 반환한다

- `handleValidationException(exception : MethodArgumentNotValidException) : ResponseEntity<Map<String, Object>>`
Request DTO의 유효성 검증 실패 예외를 처리한다.
@NotBlank, @Size, @Email 등의 Bean Validation 규칙을 만족하지 못한 경우 호출되며, 첫 번째 검증 오류 메시지를 추출하여 HTTP 400(BAD_REQUEST) 상태 코드와 함께 반환한다.

- `handleMissingRequestParam(exception : MissingServletRequestParameterException) : ResponseEntity<Map<String, Object>>`
필수 Request Parameter가 전달되지 않은 경우 발생하는 예외를 처리한다.
Controller의 @RequestParam으로 선언된 값이 요청에 포함되지 않은 경우 호출되며, 누락된 Parameter 이름을 추출하여 HTTP 400(BAD_REQUEST) 상태 코드와 함께 반환한다.

- `error(status : HttpStatus, message : String) : ResponseEntity<Map<String, Object>>`
  표준 오류 응답 객체를 생성하는 내부 보조 메서드이다.
  오류 발생 시각, 상태 코드, 메시지를 포함한 Json 응답을 구성한다.

# 4. State machine diagram
<img width="1435" height="938" alt="image" src="https://github.com/user-attachments/assets/75d53568-363c-4cdd-a1f0-c27e93d2e250" />

Creation Time은 웹 기반 애플리케이션이므로 State Machine Diagram은 사용자가 현재 접근하고 있는 페이지를 하나의 상태로 정의하고, 사용자 관점에서의 화면 전환과 기능 수행 과정을 모델링하였다.

시스템은 사용자가 최초 접속 시 Index 상태로 진입한다. Index 상태에서는 RecruitPost 목록을 조회할 수 있으며, 회원가입을 위해 Signup 상태로 이동하거나  RecruitPost를 선택하여 RecruitDetail 상태로 이동할 수 있다. 또한 로그인을 수행할 수 있으며, 로그인에 성공한 사용자가 팀에 소속되어 있는 경우 Team 상태로 전이된다.
Team 상태는 팀 활동의 중심 상태로 정의하였다. 사용자는 Post를 조회하거나 작성할 수 있으며, Post 상세 조회를 위해 PostDetail 상태로 이동할 수 있다. 또한, TeamLeader의 경우 RecruitPost을 작성하기 위해 RecruitWrite 상태로 이동할 수 있으며,  Manage 상태로 이동하여 팀 관리 기능을 수행할 수 있다. Vote, Schedule 작성 및 삭제,  RecruitPost등 다른 TeamLeader기능들도 모두 이용가능하다. 기본적으로 모두 페이지로 이동하는 이벤트에만 권한을 조건으로 작성하였으며, 해당 상태의 내부의 동작은 권한 검증을 이번 diagram에서는 생략하였다. 추가로, Vote, Schedule은 다른 페이지가 아닌 Team에서 폼을 통해 생성하는 형태지만, 페이지의 변화가 존재하고 명확한 구분을 위해 별도의 상태로 작성하였다. 생성이 완료되거나 취소되면 Team 상태로 복귀한다.

Manage 상태에서는 TeamMember와 Application 관리를 수행할 수 있다. TeamMember의 역할 변경,  TeamLeader 위임, TeamMember 제거, Application 승인 및 거절, CSV 다운로드 등의 기능을 수행가능하며 해당 작업들은 모두 manage상태에서 처리된다. 또한 TeamMember 팀 해체 기능을 수행할 수 있으며, 팀 해체가 완료되면 Index 상태로 전이된다.
PostDetail 상태에서는 Post 내용을 조회할 수 있으며 Comment 작성 및 삭제가 가능하다. Post 작성자인 경우Post 수정을 위해 WritePost 상태로 이동할 수 있으며,Post 삭제 또한 수행할 수 있다. 작업 완료 후에는 Team 상태로 복귀한다.
RecruitWrite 상태에서는 모집 분야, 모집 인원, 요구 역할, 모집 마감일 등  RecruitPost 정보를 작성할 수 있다. 또한 모집 상태를 게시 또는 종료 상태로 변경할 수 있으며, 저장 또는 취소 시 Team 상태로 복귀한다.
각 상태 내부의 entry 활동은 해당 페이지 진입 시 수행되는 초기 화면 구성 및 데이터 로딩 과정을 의미한다. 또한 do 활동은 사용자가 해당 페이지에 머무르는 동안 수행되는 입력, 조회, 선택 등의 지속적인 상호작용을 의미한다.

# 5. Implementation requirements
## 5.1 H/W platform requirements

- CPU : Intel i3 이상
- RAM : 4GB 이상
- HDD / SSD : 10GB 이상
- Network : 서버와 클라이언트 간 통신이 가능한 환경

## 5.2 S/W platform requirements

- OS : Windows 10 이상
- Implementation Language : Java, JavaScript, HTML, CSS
- Framework : Spring Boot, Spring Data JPA
- Build Tool : Gradle
- Database : H2 또는 MySQL
- Runtime Environment : JDK 17 이상
- Browser : Chrome, Edge

# 6. Glossary

| 용어 | 설명 |
|------|------|
| Team | 하나의 주제를 가지고 활동하는 최소 활동 단위. 하나의 주제로 활동하는 최소 단위. 한명의 TeamLeader와 다수 TeamMember로 구성됨. |
| RecruitPost | TeamLeader가 RegisteredUser를 모집하기 위해 작성하는 게시물. |
| TeamPost | TeamUser가 팀의 의사소통을 위해 작성하는 게시글로 해당 TeamUSer만 접근할 수 있다. | 
| Application | 사용자가 특정 팀에 참여 의사를 밝힌 상태 또는 그 신청 내역. |
| Penalty Score | 사용자의 활동 상태를 반영하기 위한 값으로, 팀 활동 중 부적절한 행동 발생 시 증가할 수 있다. |
| ProfileInfo | 이름, 이메일, 전화번호, ID, 비밀번호로 구성된 가입에 필수적인 기초 정보. | 
| CompetencyInfo | 관심 분야, 기술 스택, 희망 역할로 구성된 팀 매칭을 위한 기술적 배경 정보. |
| TeamInfo |  팀 이름, 팀 구분, 모집 인원, 요구 기술 스택, 팀 설명으로 구성된 팀 기초정보. 팀 생성시 입력받게 된다. |
| RecuritInfo | 팀구분, 모집 분야, 모집 인원, 진행 방식, 모집 기간, 관련 링크, 상세 내용으로 구성된 RecruitPost 세부내용 | 
| VoteInfo | 투표 제목, 설명, 선택 항목, 마감 기한으로 구성된 투표의 필수 항목 |
